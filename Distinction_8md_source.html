1 Distinction
2 ===========
3 
4 There are multiple other projects which pretend to achieve full performance portability.
5 Many do not satisfy the requirement for full c++ support that is needed because of the usage of template meta-programming for method specialization to avoid runtime-polymorphism.
6 
7 
8 CUDA
9 ------
10 
11 ### Positive
12 * Full control over memory, caches and execution.
13 
14 ### Negative
15 * Not platform independent: NVIDIA GPUs are required. No execution on other manufacturers GPUs or even standard CPUs.
16 * Language extension: The nvcc compiler driver is required to understand the language constructs that are used inside the kernel methods.
17 * Not compiler independent: The back-end compiler has to be one of the supported ones.

20 OpenMP
21 --------
22 
23 ### Negative
24 * No good way to control memory hierarchies and thread interaction (shared memory).
25 * Only compiler hints, no direct control over result.

28 OpenACC
29 ---------
30 
31 ### Positive
32 * Can generate x86 and CUDA code from C++ code.
33 
34 ### Negative
35 * Compiler dependent (currently not supported by many compilers and the PGI compiler is not actively enough developed and does not have a good C++ support).
36 * Only compiler hints, no direct control over result.

39 OpenCl
40 --------
41 
42 ### Positive
43 * Hardware independent (CPUs and GPUs of nearly all vendors).
44 
45 ### Negative
46 * No full C++ support.
47 * Runtime compilation -> No direct inclusion into the source (syntax highlighting?, static analysis?, debugging?, templated kernels?)

62 PGI CUDA-X86
63 -------------
64 When run on x86-based systems without a GPU, PGI CUDA C applications use multiple cores and the streaming SIMD (Single Instruction Multiple Data) capabilities of Intel and AMD CPUs for parallelvectorized execution.
65 At run-time, CUDA C programs compiled for x86 executes each CUDA thread block using a single host core, eliminating synchronization where possible.
66 
67 ### Positive
68 * Lets you write standard CUDA code and execute it on x86.
69 
70 ### Negative
71 * Not actively developed.

81 KOKKOS
82 -------------
83 See [here](https://www.xsede.org/documents/271087/586927/Edwards-2013-XSCALE13-Kokkos.pdf)
84 [here](http://trilinos.org/oldsite/events/trilinos_user_group_2013/presentations/2013-11-TUG-Kokkos-Tutorial.pdf)
85 [here](http://on-demand.gputechconf.com/supercomputing/2013/presentation/SC3103_Towards-Performance-Portable-Applications-Kokkos.pdf)
86 and [here](http://dx.doi.org/10.3233/SPR-2012-0343).
87 Source is available [here](https://github.com/trilinos/trilinos/tree/master/packages/kokkos).
88 The project is similar to *alpaka* in the way it abstracts the kernel as templated function object.
89 It provides parallel_for, parallel_reduce, etc. similar to thrust.
90 
91 ### Positive
92 * Offers buffer views with a neutral indexing scheme that maps to the underlying hardware (row/col-major, blocking, ...).
93 
94 ### Negative
95 * License.
96 * The parameters are required to be given to the function object constructor coupling algorithm and data together.
97 * The implementation of accelerator methods (atomics, ...) is selected via macros defined by the nvcc compiler. So there is no way to select between different x86 implementations for different x86 accelerators.
