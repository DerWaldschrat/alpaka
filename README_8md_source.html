1 **alpaka** - Abstraction Library for Parallel Kernel Acceleration
2 =================================================================
3 
4 The **alpaka** library is a header-only C++11 abstraction library for accelerator development.
5 
6 Its aim is to provide performance portability across accelerators through the abstraction (not hiding!) of the underlying levels of parallelism.
7 
8 It is platform independent and supports the concurrent and cooperative use of multiple devices such as the hosts CPU as well as attached accelerators as for instance CUDA GPUs and Xeon Phis (currently native execution only).
9 A multitude of accelerator back-end variants using CUDA, OpenMP (2.0/4.0), Boost.Fiber, std::thread and also serial execution is provided and can be selected depending on the device.
10 Only one implementation of the user kernel is required by utilizing the uniform kernel interface which is used by all accelerator back-ends.
11 There is no need to write special CUDA, OpenMP or custom threading code.
12 Accelerator back-ends can be mixed within a device stream.
13 The decision which accelerator back-end executes which kernel can be made at runtime.
14 
15 The **alpaka** API is currently unstable (alpha state).
16 
17 The abstraction used is very similar to the CUDA grid-blocks-threads division strategy.
18 Algorithms that should be parallelized have to be divided into a multi-dimensional grid consisting of small uniform work items.
19 These functions are called kernels and are executed in parallel threads.
20 The threads in the grid are organized in blocks.
21 All threads in a block are executed in parallel and can interact via fast shared memory.
22 Blocks are executed independently and can not interact in any way.
23 The block execution order is unspecified and depends on the accelerator in use.
24 By using this abstraction the execution can be optimally adapted to the available accelerators.

75 Dependencies
76 ------------
77 
78 [Boost](http://boost.org/) 1.56+ is the only mandatory external dependency.
79 The **alpaka** library itself just requires header-only libraries.
80 However some of the accelerator back-end implementations require different boost libraries to be built.
81 
82 When an accelerator back-end using *CUDA* is enabled, version *7.0* of the *CUDA SDK* is the minimum requirement.
83 
84 When an accelerator back-end using *OpenMP 2.0/4.0* is enabled, the compiler and the platform have to support the corresponding *OpenMP* version.
85 
86 When an accelerator back-end using *Boost.Fiber* is enabled, the develop branch of boost and the proposed boost library [`boost-fibers`](https://github.com/olk/boost-fiber) (develop branch) are required.
87 `boost-fibers`, `boost-context` and all of its dependencies are required to be build in C++14 mode `./b2 cxxflags="-std=c++14"`.

90 Usage
91 -----
92 
93 The library is header only so nothing has to be build.
94 Only the include path (`-I` or `export CPLUS_INCLUDE_PATH=`) has to be set to `<PATH-TO-ALPAKA-LIB>/include/`.
95 This allows to include the whole alpaka library with: `#include <alpaka/alpaka.hpp>`
96 
97 Code not intended to be utilized by users is hidden in the `detail` namespace.
98 
99 If you are building with the *CUDA accelerator back-end* enabled, your source files are required to have the ending `.cu` to comply with the nvcc (NVIDIA CUDA C++ compiler) rules for code files using CUDA.
100 When the *CUDA accelerator back-end* is disabled, this is not required and a `.cpp` extension is enough.
101 To allow both use-cases, it is desirable to have both, a `.cpp` file with the implementation and a `.cu` file containing only `#include <PATH/TO/IMPL.cpp>` to forward to the implementation.
102 The build system then has to use the `.cu` files when the *CUDA accelerator back-end* is enabled and the `.cpp` files else.
103 Examples how to do this with CMake can be found in the `examples` folder.
